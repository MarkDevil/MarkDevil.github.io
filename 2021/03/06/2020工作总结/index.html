工作总结

  • 目前在kid支持用户增长（2+3），用户中心（1），课系统（2）以及在线教室音视频（3）这四个业务线的测试工作，涵盖了用户拉新，用户注册/登录，约课，上课音视频等业务场景。带领8个正式员工的测试小组。我的工作内容包括：项目人力分配，项目风险评估与进度把控，业务项目提效，技术项目开发。
    在任职期间，解决了业务团队里面top问题：
    1. 针对用户增长线上bug多，服务不稳定，需求多且杂，没有重点
      • 需求分级
      • 线上问题归类
        • 业务流程不清楚，导致未覆盖 -> 梳理业务流程
        • 前端开发不规范，同一个功能，不同项目不一致 -> 推动开发实现前端功能组件化
        • 线上问题不能实时发现并召回 -> 添加线上业务逻辑监控
        • 性能问题 -> 持续压测迭代优化
    2. 服务调用链路长，测试无法直观知道服务变更的影响面
      • 开发上下游链路系统
      • 引入上下游自动化，添加QA流程确认与周知机制
    3. 问题：
      • Socket通信protobuf交互，常规的压测工具不支持。
      • 音视频线上性能问题多，调度策略不匀导致单点服务压力过高，音视频服务线上容量无法快速评估，压测准备时间长
      • 音视频的质量无法自动评估，测试依赖SDK demo
        解决：
      • 开发灵活配置的音视频推拉流工具，支持不同场景压测
      • 与运维系统集成，实现了压力机申请，脚本准备，压力执行的一键压测能力，压测流程从原来的天级到小时级
      • 实现了基于SDK音视频的自动化体系，通过自动解析海豚和服务端日志实现了音视频质量自动评估能力
    4. 针对工单系统分配不准问题
      • 开发日志聚合平台，追踪用户轨迹
      • 配置业务规则，将工单日志自动分发给指定人

具体能力：

  1. 问题定位

    • 业务问题定位
      • 熟悉产品原型与需求文档
      • 分析产品的数据流
    • 系统问题定位
      • 后端问题：
        • 数据流相关
          • 上下游是否调通
          • 模块数据发送是否正常
          • 模块间接口-协议一致性，加解密
          • 数据字段是否复用
        • 处理逻辑
          • 资源使用
          • 程序配置
          • 数据边界
        • 系统和环境
          • 系统资源 - cpu,io,句柄
          • 环境资源 - 程序版本，内核版本
        • 程序代码相关
          -获取程序和系统信息——Strace查看系统调用，系统状态获取（ps，top，/proc/pid/*，vmstat，netstat）

  2. 项目把控

    • 需求阶段
      • 积极沟通，明确需求
      • 抛出问题，积极解决
    • 技术方案
      • 考虑架构，系统交互，数据库表（表结构、字段、索引）字段设计，可拓展性
    • 测试阶段
      • 先行接口测试
      • 重视数据库测试
    • 验收
      • 业务方pre环境验收
    • 上线
      • 线上业务逻辑监控，依赖于开发打点
      • 线上接口拨测
  3. 质量运营
    • 提测阶段把控
      • 进入市场团队之后，通过什么手段解决了质量：
        • QA做的内容:
          • 确定问题与方向
            • 缺陷的数量、缺陷的严重程度、缺陷的生成原因
          • 需求分级
          • 线上业务告警监控
        • 推动开发做的：
          • 分享流程组件化，规范参数以及功能
          • 线上核心流程的业务埋点
          • 非业务改动类需求，绿色通道自测上线
    • 质量数据
      • Sonar关注千行代码严重问题数
      • jacoco增量代码覆盖率与全量代码覆盖率
      • 冒烟用例的通过率
      • 提测打回次数

  4. 系统架构理解

    • 公司级组件VKMQ,VKREDIS,VKDS,VKSCHEDULE, VKCONFIG

  5. 工具/平台建设
    • 链路追踪
    • 数据工厂
    • 音视频压测工具

  6. 持续集成CI/持续交付CD

    • 什么是CI?

      持续集成是一种软件开发实践，希望团队中的成员频繁提交代码到代码仓库，且每次提交都能通过自动化测试进行验证，从而使问题尽早暴露和解决。

  7. 专项

    • 音视频压测
      • 为什么做，有什么收益？
      • 难点
      • 怎么实现
  8. 团队建设
